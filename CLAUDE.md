# Flavour Pairing — Project Instructions

## Backlog & feature workflow

- When the user lists feature ideas, create a GitHub issue for each one (repo: `txmnzia/flavour-pairing`) with a clear description, options, and recommended approach.
- Features are prioritised periodically. **Only implement the highest-priority open issue.** Never work ahead.
- Each feature gets its own branch: `feature/<short-name>`. Never implement on `main` directly.
- When a feature is complete, push the files to `main` using `mcp__github__push_files` (see Git workflow below).
- Branches allow easy rollback — keep them clean and focused on one issue.
- Delete the feature branch locally (`git branch -D feature/xxx`) once its changes are on `main`.

## Git workflow (IMPORTANT)

The local git proxy **blocks `git push` to `main`** with a 403. Never attempt `git push origin main`.

**Correct workflow for landing changes on `main`:**

1. Develop on a feature branch: `git checkout -b feature/xxx`
2. Commit and push the feature branch normally: `git push -u origin feature/xxx` — this works fine.
3. When the feature is complete, use `mcp__github__push_files` to write the changed files directly to `main` via the GitHub API.
4. After the MCP push, sync local main: `git fetch origin && git reset --hard origin/main`
5. Delete the local feature branch: `git branch -D feature/xxx`

**Do not** try to merge locally into `main` and push — it will fail and create diverged history.

## Git hygiene

- Commit and push after every working change.
- `web/public/pairings.json` and `web/public/pairings.db` are generated build artifacts — do not commit them.
- `web/package-lock.json` should be kept up to date and committed.

## Data branch

`feature/real-data` is kept permanently as a **data store**, not a code branch. The deploy workflow reads `pairings.json` from it at build time:
```
git show origin/feature/real-data:web/public/pairings.json
```
Never delete this branch or open a PR to merge it into `main`.

## Generating real recipe data

Use the `generate-pairings` GitHub Actions workflow (Actions tab → "Generate pairings data" → Run workflow). Provide the URL of the RecipeNLG zip from the GitHub release. The workflow commits the result to `feature/real-data`.

## Architecture notes

- **No backend.** Everything is static, deployed to GitHub Pages.
- Data lives in `web/public/pairings.json`, generated by `pipeline/generate_demo.py` (demo) or `pipeline/process.py` (real data).
- The GitHub Actions deploy workflow pulls real `pairings.json` from `feature/real-data` at build time — no need to commit it to `main`.
- Scoring uses NPMI. Multi-ingredient: average NPMI across selected, minimum coverage = `max(1, round(n * 0.5))`.
